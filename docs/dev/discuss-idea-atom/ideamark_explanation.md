# IdeaMark: 知識部品の再利用のためのデータ形式

**具体例による解説：イギリス田園アクセス権 → 富士山防災政策**

---

## 1. IdeaMarkとは何か

**過去の問題と解決の記録**を構造化し、**新しい課題**に直面したときに再利用可能にするデータ形式。

Christopher Alexander の *Pattern Language* に着想を得た設計。

### 従来の課題

- 知識がPDFレポートに閉じ込められる
- 文脈依存の解決策が一般化されにくい
- 異分野間で知識が接続されない
- AI活用のための構造がない

### IdeaMarkのアプローチ

- 問題・解決・判断を分離して記述
- 参照構造（refs）で文書間を接続
- 人間とAI双方が読み書き可能
- 操作（分割・統合・抽出）が定義可能

---

## 2. IdeaMarkの構造

```
Document
  └─ メタ情報 + refs（参照）
      │
      ▼
    Section
      └─ anchorage（読み方の指定）+ refs
          │
          ▼
        Occurrence
          └─ role（役割）+ refs
              │
              ▼
            IdeaEntity
              └─ payload + children + refs
```

### 主要概念

| 概念 | 説明 |
|------|------|
| **anchorage** | Section の読み方を指定。view（problem/solution/comparison）、phase（hypothesis/plan/outcome）、domain（legal/disaster/tourism）など |
| **role** | Section 内での Occurrence の役割。problem_core / mechanism / lesson / transfer_hypothesis / tradeoff / risk など |
| **IdeaEntity** | 意味の最小単位。中立（それ自体は意味を持たない）、再帰的（子Entityを内包可能）、atomic_state（分割停止フラグ） |

> **核心**：意味は IdeaEntity に内在せず、Section の anchorage と Occurrence の role によって読み出される時に発生する

## 3. IdeaMark文書への操作

### Breakdown（分割）

```
1文書 → 子文書 + 子文書 + ...
```

- Section / 地点 / 問題別に分割
- refs で元文書を参照保持

### Convergent（統合）

```
文書A + 文書B + ... → 統合文書
```

- 複数文書を統合・収束
- Decision Package の生成

### Extraction（抽出）

```
元文書 → 部分抽出（別ドメイン向け）
```

- 別ドメインへの転用
- **⚠️ 転用リスク Entity の生成が必須**

> **操作パラメータとしての anchorage / role**：同じ IdeaEntity 群に対して、anchorage を変えると読み出される意味が変わる。これにより、データ（Entity）とビュー（読み方）を分離できる。

## 4. 具体例①：イギリス田園アクセス権の構造化

### 出典

- **岩本純明 (1997)**「田園レクリエーションとアクセス権 —イギリスの経験」
- 農耕の技術と文化 Vol.20

### 論文の主題

イギリスにおける田園への公衆アクセス権が、150年にわたる市民運動と法制化によってどのように形成されてきたか。

### IdeaMark化の目的

「通行権管理」の制度設計パターンを抽出し、他地域・他ドメインへの転用を可能にする。

### 抽出した Section 構造

| Section ID | 内容 | anchorage |
|------------|------|-----------|
| SEC-02 | 通行権の法的構造 | institutional_analysis, legal_framework |
| SEC-03 | 1949年国立公園法 | institutional_history, national_parks |
| SEC-04 | 土地所有とアクセスの緊張 | conflict_analysis, property_rights |
| SEC-07 | Pattern Catalog | pattern, reusable |

## 5. 抽出されたパターン：Definitive Map（確定地図）

### 問題

アクセス可能なルートが不明確で、土地所有者と利用者の間で紛争が生じる。

### 解決

すべての公共通行権を地図上に記録し、法的証拠として確定する。

### 構成要素

1. 地図の作成・更新手続き
2. 異議申立・確定手続き
3. 法的効力の付与
4. 維持管理責任の明確化

### IdeaEntity としての記述（抜粋）

```yaml
id: "IE-PAT-UK-DEFINITIVE-MAP"
atomic_state: true
refs:
  - uri: "ideamark://patterns/PAT-01"
    role: "variant_of"
```

> **再利用のポイント**：このパターンは「ルートの法的確定」という抽象構造を持ち、「登山者の位置の確定」など別ドメインに転用可能。

## 6. 具体例②：富士山防災への Extraction（抽出）

### 対象問題：富士山チャレンジ（吉本ほか 2019）

御嶽山噴火（2014年）では入山者数が不明で救助活動が困難を極めた。富士山ではビーコンによる登山者動態把握の実証実験を実施。3時間で97%の位置把握が可能に。

### Extraction マッピング

| イギリス（アクセス権） | → | 富士山（防災） |
|------------------------|---|----------------|
| 通行権の確立 | → | 登山ルートの指定・管理 |
| 確定地図（definitive map） | → | 登山者位置把握システム |
| 土地所有者の維持義務 | → | 山小屋の情報インフラ義務 |
| 市民の通行権 | → | 登山者の権利と責任 |

> **抽出のロジック**：「ルートの法的確定」→「登山者位置の確定」、「土地所有者の義務」→「山小屋の義務」という類推的マッピング。`refs.role = "analogous_pattern"` で明示。

## 7. 転用リスクの構造化（必須Entity）

### 設計原則

> **異ドメインへの Extraction（抽出）操作では、転用リスクを記述した IdeaEntity の生成を必須とする**

### リスクカテゴリ

#### 1. 文脈の相違

- **イギリス**：土地所有権 vs 公共的利用権（水平的対立）
- **富士山**：登山者の自由 vs 公的安全管理（垂直的対立）
- → 正当化根拠が異なる

#### 2. 時間スケールの相違

- **イギリス**：150年の漸進的形成
- **富士山**：噴火リスクは待ってくれない
- → 災害を契機に短期間での制度化が求められる可能性

#### 3. 技術的前提の相違

- **イギリス**：紙の地図、物理的標識
- **富士山**：デジタル追跡、リアルタイム通信
- → 技術の信頼性・持続性への依存

#### 4. プライバシー問題

- イギリスの通行権は位置追跡を含まない
- 富士山のビーコンは個人の移動を追跡
- → 目的外利用の防止、データ管理の厳格化が必要

### メタデータ表現

```yaml
extraction_metadata:
  transfer_type: "analogical"
  confidence: "hypothesis"
  requires_validation:
    - 法的実現可能性
    - 山小屋ヒアリング
    - コスト試算
```

## 8. 生成された転用仮説パッケージ

### 仮説1：登山者登録・追跡の制度化

- 富士山チャレンジを制度化の基盤に
- 段階的に任意→推奨→義務化
- 登録データの法的位置づけを明確化

### 仮説2：山小屋の防災インフラ義務

- 受信機設置を営業条件に組み込む
- 費用は公的補助と入山料で分担
- データ提供・情報伝達義務を明文化

### 仮説3：権利と責任のセット化

- 「登山する権利」と「位置情報提供の責任」
- 「救助を受ける権利」と「登山届提出の責任」
- 行動規範として社会的合意を形成

### 仮説4：漸進的制度形成

- 実証実験→推奨運用→条例化→法制化
- 災害を契機とした「窓」を活用
- 長期的な社会的合意形成を並行

### 評価基準（KPI）

| 基準 | 目標 |
|------|------|
| 網羅性 | 登山者の90%以上を把握 |
| 即時性 | 30分で80%以上の位置把握 |
| 持続性 | 通年・継続的運用 |
| 受容性 | 社会的規範として定着 |

> **次のステップ（Convergent）**：この抽出文書を既存の富士山施策文書（SEC-KNG-FUJI-02）と統合し、Timed Entry との組み合わせを検討

## 9. まとめ

### IdeaMarkの核心

1. **意味は Entity に内在せず、読み方（anchorage/role）で発生**
2. **データとビューの分離**により再利用性を確保
3. **操作（Breakdown/Convergent/Extraction）が定義可能**
4. **異ドメイン転用には転用リスク Entity が必須**

### 今回の実証

1. イギリス論文 → IdeaMark文書化
2. パターン抽出（Definitive Map等）
3. 富士山防災への Extraction
4. 転用仮説パッケージの生成

# 10 — Status, Versioning, and Regeneration Model

**Part:** 3 — Core Model  
**Status:** Specification Draft  
**Version:** IdeaMark Core v1.2.0

## 10.1 Purpose

This section defines how status, versioning, and regeneration are treated in the IdeaMark Core Model.

An IdeaMark document is an operational snapshot produced by Projection-guided Decomposition.

Because Original Sources, Projections, tools, profiles, and review judgments may change, the model must allow documents to be drafted, reviewed, regenerated, superseded, deprecated, compared, and reused without requiring global identity stability or final meaning.

## 10.2 Operational Snapshot Definition

An IdeaMark document is an operational snapshot of a Decomposition result.

It records that, at a certain time and under a certain Projection Context, certain Original Source material was structured into Sections, Occurrences, Entities, anchors, metadata, and optional structure.

A snapshot is not:

- a permanent semantic truth;
- the only valid model of the source;
- a complete representation of the source;
- a guarantee that future reconstructions will use the same structures;
- a substitute for Original Source material.

Multiple snapshots may coexist.

## 10.3 Status Definition

Status is operational metadata indicating the current usability, review state, or lifecycle condition of an IdeaMark document or structure.

Part 3 primarily defines document-level status.

Object-level status for Sections, Occurrences, or Entities may be allowed by profiles or implementations, but Core should define document-level status first.

Status supports:

- review;
- safe reuse;
- regeneration;
- deprecation;
- comparison;
- tool interoperability;
- human interpretation of document maturity.

## 10.4 Document-level Status

Document-level status may indicate whether an IdeaMark document is:

- draft;
- generated;
- reviewed;
- approved;
- experimental;
- partial;
- deprecated;
- superseded;
- failed;
- archived.

Part 3 does not require this exact vocabulary.

Part 4, profiles, repositories, or implementations may define concrete status values.

At the conceptual level, status should help later users and tools judge how the document should be handled.

## 10.5 Object-level Status

Sections, Occurrences, and Entities may also need status in some contexts.

For example:

- an inferred Entity may need review;
- a Section anchor may be stale;
- an Occurrence role may be disputed;
- a generated Entity may be deprecated but retained for traceability;
- a Section may be superseded by a regenerated Section.

Core does not require object-level status.

It should allow profiles or implementations to add it when needed.

## 10.6 Versioning Definition

Versioning records which specification, source, Projection, generator, profile, or document revision is involved in an IdeaMark document.

Versioning supports later comparison and reconstruction.

Versioning may apply to:

- IdeaMark Core specification version;
- document revision;
- Original Source revision;
- Projection revision;
- generator version;
- profile version;
- schema or validation version;
- review revision;
- derived document revision.

Part 3 defines the conceptual need for versioning.

Part 4 defines concrete representation.

## 10.7 Specification Version

An IdeaMark document should identify the IdeaMark Core version or profile version it follows.

This allows later tools to understand which model assumptions and validation rules may apply.

For example:

```yaml
meta:
  spec_version: 1.2.0
```

This example is illustrative, not normative YAML for Part 3.

## 10.8 Document Identity and Revision

An IdeaMark document should have an identity sufficient for local or repository-level reference.

Core does not require a universal global document identity system.

Document identity may be:

- locally assigned;
- repository scoped;
- URI-based;
- content-hash-based;
- implementation-defined;
- generated by an authoring tool.

Document revision may be tracked by:

- version number;
- content hash;
- Git commit;
- timestamp;
- generation run ID;
- implementation-defined revision marker.

## 10.9 Source Revision

Original Sources may change over time.

A document should record source revision information when it matters for reconstruction or review.

Source revision may include:

- Git commit hash;
- document version;
- timestamp;
- content hash;
- dataset snapshot;
- media file version;
- archival locator;
- implementation-defined revision ID.

Anchors should be interpreted relative to source revision when available.

## 10.10 Projection Revision

Projection Context may also change over time.

A document should record Projection revision information when the Projection is versioned or externally referenced.

Projection revision may include:

- Projection ID;
- Projection version;
- Projection library reference;
- file path and revision;
- inline Projection content hash;
- generation-time snapshot;
- compatibility note.

This helps later systems decide whether a document should be reused, regenerated, or compared with a newer Projection.

## 10.11 Generator and Tool Version

If a document is generated or assisted by tools, generator information may be useful.

This may include:

- tool name;
- tool version;
- model name;
- model version;
- prompt or workflow reference;
- generation run ID;
- configuration hash;
- authoring profile.

Part 3 does not require all generator details.

However, generator metadata can improve review, reproducibility, and debugging.

## 10.12 Regeneration Definition

Regeneration is the creation of a new or updated IdeaMark document from source material and Projection Context after some relevant change.

Regeneration may be triggered by changes in:

- Original Source material;
- Projection Context;
- generator or tool behavior;
- authoring profile;
- review feedback;
- granularity requirements;
- reconstruction needs;
- specification version;
- validation rules.

Regeneration is not assumed to produce the same Sections, Occurrences, or Entities.

## 10.13 Regeneration Is Not In-place Truth Correction

Regeneration should not be understood as correcting a document toward one final true structure.

Because IdeaMark documents are Projection-shaped snapshots, a regenerated document may be different but still coexist with the earlier document.

A regeneration may:

- preserve some structures;
- split Sections;
- merge Sections;
- create new Entities;
- remove Entities;
- change Occurrence roles;
- update anchors;
- record different Projection coverage;
- produce a separate document rather than overwrite the old one.

Core should allow coexistence.

## 10.14 Supersession and Deprecation

A document may be marked as superseded or deprecated.

Supersession means another document is preferred for some use.

Deprecation means the document should no longer be used except for traceability, history, or review.

Supersession and deprecation do not imply that the old document never had value.

They indicate operational status.

A deprecated or superseded document may still be an Original Source for historical analysis.

## 10.15 Comparison Across Versions

Versioning and status should support comparison across documents.

A comparison may ask:

- What changed in the Original Source?
- What changed in the Projection?
- What changed in Section boundaries?
- What changed in Occurrence roles?
- What changed in Entity material?
- Which anchors became stale?
- Which document is preferred for a given reconstruction task?

Part 3 does not define comparison algorithms.

It defines the metadata responsibilities that make comparison possible.

## 10.16 Identity Stability Across Regeneration

Core does not require identity stability across regeneration.

A regenerated document may preserve IDs when useful, but this is not mandatory.

IDs for Sections, Occurrences, and Entities are primarily reference labels unless a profile defines stronger rules.

Implementations may choose to preserve IDs to support diffs and review.

However, forcing stable IDs may create false continuity when the Projection-shaped structure has changed substantially.

## 10.17 Partial, Draft, and Failed Documents

A document may be partial, draft, or failed.

Such documents may still be useful during authoring, review, or experimentation.

However, their status should be clear.

A partial document may preserve useful source windows even if it does not cover all intended material.

A failed document may preserve diagnostic information but should not be treated as a reliable access structure.

## 10.18 Review

Review is outside Core as a workflow, but status and versioning should support review.

A reviewer may check:

- whether source references are valid;
- whether anchors return to appropriate source material;
- whether Sections are Projection-shaped local source windows;
- whether Occurrence roles are useful;
- whether Entity payloads overstate the source;
- whether metadata records relevant Projection Context;
- whether regeneration is needed.

Part 3 does not define who reviews or how review is performed.

## 10.19 Status and Authority

Status does not equal truth or authority.

A reviewed or approved document is operationally marked as reviewed or approved under some process.

Part 3 does not define that process.

Authority, trust, legal validity, institutional approval, and governance belong to profiles, repositories, institutions, or companion specifications.

Core only defines that status can be recorded and used operationally.

## 10.20 Document as Historical Artifact

Older IdeaMark documents may remain useful.

They may record:

- earlier source states;
- earlier Projection assumptions;
- earlier decomposition decisions;
- historical review context;
- deprecated but relevant interpretations;
- comparison material for future analysis.

Therefore, deletion is not the only response to obsolescence.

Archiving, deprecation, or supersession may be preferable.

## 10.21 Conceptual Minimum

At the conceptual level, status and versioning should allow later users or tools to answer:

```text
What is this document?
Which Core version or profile does it follow?
What source revision did it use?
What Projection revision shaped it?
What is its operational status?
Has it been superseded, deprecated, reviewed, or regenerated?
Can it be safely reused for this reconstruction task?
```

Part 4 may define fields such as:

```yaml
meta:
  document_id:
  spec_version:
  status:
  generated_at:
  generator:
  sources:
    - id:
      revision:
  projections:
    - role:
      ref_id:
      revision:
  supersedes:
  superseded_by:
```

This example is illustrative, not normative YAML for Part 3.

## 10.22 Weak Status or Versioning

A document may be weak if it:

- has no status despite being draft, partial, or generated;
- lacks source revision information where anchors depend on revision;
- lacks Projection revision information where Projection changed over time;
- cannot be compared to regenerated versions;
- uses stable IDs to imply continuity where structures changed substantially;
- claims approval without recording what approval means in the relevant profile or repository.

Part 4 or profiles may define validation levels.

Part 3 identifies these conceptual risks.

## 10.23 Invariants

Status, versioning, and regeneration should preserve the following invariants:

1. An IdeaMark document is an operational snapshot.
2. Status is operational metadata, not final truth.
3. Document-level status is Core; object-level status is optional or profile-defined.
4. Source revision should be recorded when relevant to reconstruction.
5. Projection revision should be recorded when relevant to reconstruction.
6. Regeneration may produce a different valid document.
7. Multiple regenerated documents may coexist.
8. ID stability across regeneration is not required by Core.
9. Superseded or deprecated documents may remain useful for traceability and history.
10. Review workflow and governance are outside Core.

## 10.24 Summary

Status, versioning, and regeneration keep IdeaMark operational without turning it into a final truth system.

They allow documents to be reviewed, compared, regenerated, deprecated, and reused while preserving the Core stance that IdeaMark documents are Projection-shaped access structures rather than final meanings or source replacements.

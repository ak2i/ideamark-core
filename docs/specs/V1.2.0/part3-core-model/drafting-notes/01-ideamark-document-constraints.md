# IdeaMark Document Constraints

**Part:** 3 — Core Model  
**Status:** Design Note  
**Type:** Constraint Set / Pre-Modeling Notes

This document records the constraints that an IdeaMark Document should satisfy before deciding the final internal structure of the Core Model.

The purpose of this note is to preserve the design requirements independently from any current structural solution such as Section, Occurrence, Entity, Anchorage, metadata, or future replacements.

## 1. Projection-guided generation

An IdeaMark Document is not generated by general extraction from an Original Source.

It is generated from Original Source material under a Projection Context.

```text
Original Source material x Projection Context -> IdeaMark Document
```

Projection-independent generation may be technically possible, but it is not the intended model.

## 2. Separation from Original Source

An IdeaMark Document is separate from the Original Source material from which it was generated.

It does not replace the Original Source.

It must preserve enough traceability so that later humans, AI systems, or tools can return to relevant Original Source material when needed.

## 3. No stored final meaning

An IdeaMark Document does not store final meaning.

Meaning is treated as something that becomes observable when expression, interpretation, Situation, and activity meet and produce intellectual, practical, or observational consequences.

Therefore, the document should not attempt to encode meaning as a finished object.

## 4. Support for Human-AI Intellectual Activity

The purpose of an IdeaMark Document is not knowledge preservation by itself.

Its purpose is to help future humans and AI systems begin, continue, or improve Human-AI Intellectual Activity.

The document should therefore preserve structures that can function as materials for later activation, reconstruction, and use.

## 5. Reconstruction support

An IdeaMark Document should support reconstruction.

It should help later processes retrieve Original Source material, apply a Projection or compatible Projection, and produce expressions or actions that can activate useful intellectual activity.

The document is not expected to complete the intellectual activity by itself.

## 6. Structure rather than meaning

The document should preserve reusable structure rather than final meaning.

The structure should be sufficient to reduce interpretation cost and to allow later reconstruction from Original Sources through Projections.

## 7. Projection is external but referable

Projection is not part of the Core Model in the strict sense.

An IdeaMark Document should be able to refer to the Projection or Projection Context that shaped generation.

It may also record inline Projection material when useful for interoperability or later reconstruction.

## 8. Multiple Original Sources

An IdeaMark Document may be generated from multiple Original Sources.

The Core Model should not assume a one-document-to-one-source relationship.

## 9. Multiple Projections

An IdeaMark Document may be generated under multiple Projections or a composite Projection Context.

The Core Model should not assume a one-document-to-one-projection relationship.

## 10. Generation Projection and use Projection need not match

The Projection used during generation and the Projection used later during retrieval, matching, filtering, or reconstruction do not need to be identical.

Reuse may occur through partial match, approximate match, inclusion, specialization, compatible intent, or transformable assumptions.

## 11. Media-independent Original Sources

Original Sources are not limited to text documents.

Documents, images, audio, video, source code, datasets, sensor logs, interaction histories, generated artifacts, streams, composites, and future media may all become Original Sources when they can participate in Projection-guided Human-AI Intellectual Activity.

Practical use requires that humans, AI systems, or supporting tools can observe, reference, transform, or input the source material sufficiently for Decomposition and later reconstruction.

## 12. Algorithm and workflow non-prescription

The Core Model should not prescribe how an IdeaMark Document is generated.

Generation may be performed by humans, LLMs, tools, POR-like processes, future AI systems, or mixed workflows.

Core should define the conceptual boundary, not the execution mechanism.

## 13. Decomposition is not fully reversible

The Decomposition process that produced an IdeaMark Document is not assumed to be fully reconstructable from the document.

The document may record local rationale, generation notes, method notes, or explanatory traces, but these are partial explanations rather than a complete inverse mapping.

## 14. Traceability is required

An IdeaMark Document must preserve enough traceability to support return to relevant Original Source material.

Traceability is more important than semantic completeness.

The document should make it possible to understand what material the generated structure is anchored to, based on, or meant to reopen.

## 15. Interpretation cost reduction

The document should reduce interpretation cost.

This includes the cost of creation, review, maintenance, handoff, retrieval, reconstruction, education, and reuse.

A structure that is formally expressive but increases interpretation cost may be contrary to the Core design intent.

## 16. Future-proofing

The Core Model should remain open to future AI systems, source media, Projection practices, reconstruction methods, and activation expressions.

It should define minimal structural responsibilities and avoid freezing domain-specific intellectual activity.

## 17. Minimal Core

The Core should specify as little as possible while still enabling traceable reconstruction and interoperability.

Concepts that are not clearly necessary should remain metadata, extensions, implementation choices, or later specifications.

## 18. Recursive use

An IdeaMark Document may itself become an Original Source for later IdeaMark generation.

This does not make IdeaMark a knowledge base.

It means that IdeaMark documents, like other artifacts, can participate as material in later Projection-guided Human-AI Intellectual Activity.

## 19. Operational snapshot

An IdeaMark Document is an operational snapshot.

It represents the result of a Projection-guided structuring act at a particular time, not a final or universal representation of the Original Source.

Multiple valid IdeaMark Documents may coexist for the same source material.

## 20. Human and AI participants

The Core Model should support both human and AI participants.

It should not assume that interpretation, reconstruction, review, or activation is performed only by humans.

However, it should also preserve enough structure and traceability for human inspection, review, and correction.

## 21. Structural solution remains open

The constraints above do not by themselves require Section, Occurrence, Entity, Anchorage, Relation, Perspective, or Provenance as final Core Model objects.

Those are design solutions, not constraints.

Part 3 should use this constraint set to evaluate whether the current structural solution is sufficient, whether it should be simplified, or whether a replacement structure is needed.

## 22. Design implication

The primary question for the remaining Core Model design is:

> What minimal structure must an IdeaMark Document preserve so that future humans and AI systems can retrieve Original Source material, apply a Projection or compatible Projection, and generate expressions that activate useful Human-AI Intellectual Activity?

This question should guide the finalization of Part 3.
